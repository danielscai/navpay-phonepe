# 步骤 4：HTTPS 请求拦截 — 分析与计划

日期：2026-01-30

## 背景
步骤 4 目标是在**模拟器中真实验证 HTTPS 请求拦截**。当前需要的形态是**注入到合并后的 PhonePe APK**，拦截器无需 UI 展示，通过日志服务器接收并验证拦截结果。

## 当前代码结构梳理
- Demo App：`src/https_interceptor/`
  - UI 仅用于演示；最终验证以**注入 APK + 日志服务器**为准
- 日志服务器：`src/log_server/`
  - `POST /api/log` 接收日志，SQLite 持久化 + Web UI
- 注入脚本：`tools/patch_https_interceptor.sh`
  - 向目标 APK 注入 `RemoteLoggingInterceptor` 并尝试修改 `HookUtil` 注入链路

## 发现的问题/风险点
1. **注入脚本检查文件不一致**
   - `patch_https_interceptor.sh` 只生成 `RemoteLoggingInterceptor.smali`，但校验 `RemoteLoggingInterceptor$1.smali`，会导致误报。
2. **HookUtil 找不到时缺少真正的调用路径**
   - 需要改为注入 `OkHttpClient$Builder.build()` 或其他稳定入口，确保拦截器实际生效。
3. **JAVA_HOME 硬编码路径**
   - 脚本固定 `openjdk@17` 路径，可能与环境不一致。
4. **默认日志地址依赖 adb reverse**
   - `http://127.0.0.1:8088` 需要 `adb reverse`，需要在测试流程中保证执行。

## 计划修复项
- 修复脚本校验逻辑（去掉不存在的 `$1` smali 检查或补齐生成）。
- 当 `HookUtil` 找不到时，补充**可执行入口注入**（明确 hook 调用链，确保 interceptor 实际生效）。
- 统一 `JAVA_HOME` 与运行环境（自动检测或使用系统默认）。
- 编写**步骤 4 专用验证脚本**：清理旧产物 → 注入 → 重新打包 → 签名 → 安装 → 运行 → 日志验证（以服务端日志为准）。

## 计划执行流程（真实验证）
1. 启动 `log_server`（含 `adb reverse`）。
2. 反编译目标 APK（来自步骤 2 合并产物）。
3. 执行注入脚本，确保拦截器被真正挂载到 OkHttp。
4. 重打包、签名并安装到模拟器。
5. 启动应用并触发网络请求。
6. 在服务器 Web UI/日志中验证**请求被拦截并上报**（无需任何 UI）。

## 通过标准
- 模拟器中应用可正常启动。
- 日志服务器收到至少 1 条真实请求日志（含 URL / method / statusCode）。
- 若启用 token 检测，日志中能看到 `token_detected` 字段。
