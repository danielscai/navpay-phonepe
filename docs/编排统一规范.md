# 编排统一规范（Orchestrator / Modules / Cache）

适用范围：`src/pipeline/orch`、`src/*` 模块脚本、根目录 `cache/` 与 `package.json`。

目标：后续新增模块和流程变更必须遵循同一套命名、目录与执行约束，避免再次出现多套入口与兼容壳。

## 0. src 分层目录规范

`src/` 必须按用途分层，禁止把模块代码、编排代码、服务端代码平铺在同一层：

- `src/apk/`：APK 注入相关模块与共享框架
  - 例如：`src/apk/signature_bypass`、`src/apk/https_interceptor`、`src/apk/phonepehelper`、`src/apk/_framework`
- `src/pipeline/`：编排与流程工具
  - 例如：`src/pipeline/orch`、`src/pipeline/tools`
- `src/services/`：服务端/管理端项目
  - 例如：`src/services/log_server`、`src/services/navpay_server`、`src/services/navpay-admin`

## 1. 统一入口

- 唯一编排入口：`python3 src/pipeline/orch/orchestrator.py`
- 禁止新增平行入口（例如新的 `cache_manager.py`、模块级主流程 CLI）。

## 2. 流程命名统一

- 主流程动作统一为：
  - `plan`
  - `pre-cache`
  - `compile-modules`
  - `merge`
  - `compile`
  - `test`
- 术语统一：
  - 不再使用 `build-modules`，统一为 `compile-modules`
  - 不再使用 `inject` 作为主流程动作，统一为 `merge`
  - 模块脚本层也统一使用 `compile.sh` / `merge.sh`

## 3. package.json 命名空间规范

- 主流程命名空间：`flow:*`
  - 例如：`flow:plan`、`flow:smali`、`flow:merge`、`flow:apk`、`flow:test`
- 运维/诊断命名空间：`orch:*`
  - 例如：`orch:graph`、`orch:status`、`orch:reset`、`orch:rebuild`
- 禁止恢复旧兼容别名命令（`sigbypass` / `https` / `phonepehelper` / `profile <name> <action>`）。

补充：

- `flow:compile` 允许保留为 `flow:apk` 的别名，但默认必须走缓存复用。
- 需要全量重建时使用显式参数或独立命令（例如 `flow:compile:fresh` / `flow:apk:fresh`）。

## 4. 模块脚本结构规范（src/*）

每个参与 orchestrator 的模块只保留两类主脚本：

- `scripts/compile.sh`
- `scripts/merge.sh`

约束：

- `merge.sh` 必须是 artifact-only：
  - 必须接收 `--artifact-dir`
  - 不允许在 `merge` 阶段隐式触发编译
- 禁止模块内保留历史薄封装兼容脚本（如 `build_artifacts.sh`、`inject.sh` 的兼容壳）。

## 5. cache 目录规范

### 5.1 PhonePe 基线链路缓存

统一放在：

- `cache/phonepe/from_device`
- `cache/phonepe/merged`
- `cache/phonepe/decompiled`

禁止再使用：

- `cache/phonepe_from_device`
- `cache/phonepe_merged`
- `cache/phonepe_decompiled`

### 5.2 Profile 缓存

统一放在：

- `cache/profiles/<profile>/workspace`
- `cache/profiles/<profile>/build`

禁止再使用扁平目录：

- `cache/profile_<profile>_workspace`
- `cache/profile_<profile>_build`

## 6. Manifest 规范（src/pipeline/orch/cache_manifest.json）

模块定义必须包含：

- `builder.command` 指向 `scripts/compile.sh`
- `merge_script` 指向 `scripts/merge.sh`
- `fingerprint_inputs` 只声明真实影响编译/合并结果的输入
- `outputs` 只声明 orchestrator 消费的产物

## 7. 兼容策略

- 当前策略是“清理优先，不保留 compatibility wrapper”。
- 新变更默认直接升级到主规范，不新增“旧命令继续可用”的过渡层。

## 8. 变更门禁

涉及编排/脚本/cache 结构改动时，至少执行：

- `python3 -m unittest discover -s src/pipeline/orch/tests -p 'test_*.py' -v`
- `python3 src/pipeline/orch/orchestrator.py plan`

并确保：

- `package.json` 中 `flow:*` 与 `orch:*` 分组仍清晰
- 文档（`src/README.md`、`src/pipeline/orch/README.md`）路径与命令保持同步。
