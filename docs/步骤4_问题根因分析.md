# 步骤4 注入崩溃根因分析（供 review）

## 结论（先看）
- **崩溃根因不是“原版 okhttp3 被混淆”**，而是：
  1) 运行时实际使用的 `okhttp3` 类在 **classes2.dex** 中，方法名已被缩写（如 `Request.a/b/c/d`、`Chain.a()`），
  2) 我们的拦截器调用了 **标准 API**（如 `Request.url()`、`Chain.proceed()`），导致 `NoSuchMethodError`。
- 当拦截器逻辑被最小化（仅调用 proceed 并做反射兜底）后，**app 可正常进入登录界面**，说明注入链路本身没问题，**问题出在拦截器内部调用的 API 与运行时 okhttp3 不匹配**。

## 证据链
### 1) 崩溃日志（AndroidRuntime）
- 初始崩溃（示例）：
  - `NoSuchMethodError: Request.url()`
  - 发生在 `RemoteLoggingInterceptor.intercept()`
- 后续精简后出现：
  - `NoSuchMethodError: Interceptor$Chain.proceed(Request)`

### 2) 运行时 okhttp3.Request 实际定义
- 来自注入后的 APK 解包目录：
  - `temp/https_interceptor_test/base_decompiled/smali_classes2/okhttp3/Request.smali`
- 该类没有 `url()/method()/headers()/body()`，只有缩写字段：
  - `a: HttpUrl`
  - `b: String`
  - `c: Headers`
  - `d: RequestBody`
- 说明 **运行时不是标准 API**。

### 3) 运行时 okhttp3.Interceptor$Chain 实际定义
- `temp/https_interceptor_test/base_decompiled/smali_classes2/okhttp3/Interceptor$Chain.smali`
- 只有：
  - `a(Request): Response`
  - `request(): Request`
- **没有 `proceed(Request)`**。

### 4) 拦截器重复类导致旧版本被加载（已修复）
- 早期发现注入后存在多份 `RemoteLoggingInterceptor.smali`（多个 smali_classes*），
  导致运行时加载旧版本（仍调用 `Request.url()`）。
- 修复方式：在注入脚本中 **先清理所有 `com/httpinterceptor` 再复制新 smali**。

## 关键推导
1. 原版 APK 的 okhttp3 未混淆、版本为 4.10.0（已写入 README）。
2. 但运行时实际加载的是 **classes2.dex 中的缩写版 okhttp3**。
3. 拦截器使用标准 API → `NoSuchMethodError` → 崩溃。
4. 拦截器最小化后不崩溃，说明注入链路正确。

## 处理策略（已验证）
- **最小化拦截器**（仅 `chain.request()` + `chain.proceed()`），并对 `proceed` 做反射兜底，验证注入链路 OK。
- 后续若要逐步恢复功能，必须基于运行时的缩写方法/字段做兼容适配。

## 后续建议（待确认）
- 若恢复请求/响应读取：
  - `Request.url()` → 读取字段 `a`
  - `Request.method()` → 读取字段 `b`
  - `Request.headers()` → 读取字段 `c`
  - `Request.body()` → 读取字段 `d`
- `Chain.proceed()` → `Chain.a()`
- 以上适配可以通过：
  1) 反射兜底，
  2) 注入阶段对拦截器 smali 做自动映射。

## 结论补充
- **注入链路 OK，崩溃来自 API 不匹配**。
- 必须优先保证拦截器调用与运行时 okhttp3 一致。
